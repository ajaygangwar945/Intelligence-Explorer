INTELLIGENCE EXPLORER - FULL PROJECT TECHNOLOGY, CONCEPTS, LOGIC DOCUMENT

Project Name:
Intelligence Explorer

Live Deployment:
https://intelligence-explorer.vercel.app/

Project Type:
Static educational web application with optional Node.js utility script for PDF-to-text extraction.

====================================================================
1) PROJECT GOAL
====================================================================
This project is built to present AI learning topics in a visually engaging dashboard and detailed module pages.
It combines:
- Strong visual design
- Smooth UI animations
- Topic-wise educational content
- Lightweight client-side 3D backgrounds using Three.js
- Simple static deployment on Vercel

====================================================================
4) END-TO-END PROJECT LOGIC FLOW
====================================================================

1. User opens deployed URL.
2. Vercel rewrite serves dashboard.html as the homepage.
3. Dashboard loads styles/dashboard.css, favicon, and Three.js background scene.
4. User clicks one of six topic cards.
5. Browser opens corresponding module page in pages/.
6. Module page displays theory content plus themed Three.js animation.
7. User can return using back link to ../dashboard.html.
8. Optional content workflow: extract_pdfs.js reads course PDFs and writes text files into pdf_analysis/.

====================================================================
5) FILE-BY-FILE LOGIC, WHERE USED, WHY USED
====================================================================

A) dashboard.html
Where used:
- Main entry page.

Main logic:
- Defines app identity (title, favicon, heading).
- Renders six topic cards linking to module pages.
- Loads Three.js and builds animated background:
  - Creates scene, camera, renderer
  - Creates wireframe icosahedron sphere
  - Creates point cloud node system
  - Animates rotation and handles resize events

Why used:
- Acts as central navigation and first impression of project.


B) styles/dashboard.css
Where used:
- Linked in dashboard.html.

Main logic:
- Defines color tokens in :root.
- Controls layout (container, grid, typography, cards).
- Adds hover, staggered card animations, icon animations.
- Provides responsive rules for smaller screens.

Why used:
- Keeps dashboard styling maintainable and reusable.


C) pages/cnn.html
Where used:
- CNN and Deep Learning learning module.

Main logic:
- Topic content sections and bullet explanations.
- Three.js side-canvas with animated layered planes (represents CNN layers).
- Resize-aware rendering.

Why used:
- Connects CNN concept to an appropriate visual metaphor (layered depth).


D) pages/genai.html
Where used:
- Generative AI learning module.

Main logic:
- Explains GAN, VAE, Diffusion, and ethics.
- Three.js particle field for latent-space concept.
- Mouse movement influences particle position for interactivity.

Why used:
- Demonstrates creativity and high-dimensional generative space visually.


E) pages/intro.html
Where used:
- AI and ML foundations module.

Main logic:
- Covers supervised/unsupervised/reinforcement learning and ML lifecycle.
- Three.js node-link graph generated by distance checks between nodes.
- Group rotates continuously.

Why used:
- Visualizes interconnected foundational concepts and workflow relationships.


F) pages/llm.html
Where used:
- Large Language Models module.

Main logic:
- Describes transformers, attention, pretraining, fine-tuning.
- Three.js moving wireframe boxes in depth (data tunnel effect).
- Resets box depth position to loop animation.

Why used:
- Symbolizes continuous flow of tokens/data through deep model layers.


G) pages/nlp.html
Where used:
- NLP module.

Main logic:
- Explains NLU/NLG, preprocessing pipeline, sentiment/dialog systems.
- Three.js orbiting spheres around center (word cluster metaphor).
- Rotation on x and y axes for dynamic motion.

Why used:
- Represents semantic clusters and language structure dynamics.


H) pages/prompt.html
Where used:
- Prompt Engineering module.

Main logic:
- Covers few-shot, chain-of-thought, priming/roleplay, quality impact.
- Three.js octahedron wireframe rotating prism.
- Right-side canvas for balanced content + visual layout.

Why used:
- Illustrates precision and structure in prompt crafting.


I) extract_pdfs.js
Where used:
- Local utility script (not browser runtime).

Main logic:
- Creates output directory if missing.
- Loops through fixed PDF list.
- For each PDF:
  - Load file with PDFParser
  - Handle parser errors and continue
  - Extract raw text
  - Save as matching .txt file in pdf_analysis/

Why used:
- Automates text extraction from course source documents.


J) pdf_analysis/*.txt
Where used:
- Data output folder for extracted textual content.

Main logic:
- Stores plain text versions of each source module PDF.

Why used:
- Makes content searchable, analyzable, and easier to reuse.


K) package.json
Where used:
- Project dependency declaration.

Main logic:
- Defines pdf2json and pdf-parse versions required for extraction utility.

Why used:
- Ensures reproducible environment setup.


L) vercel.json
Where used:
- Vercel deployment routing configuration.

Main logic:
- Rewrite rule: "/" -> "/dashboard.html"

Why used:
- Root domain opens dashboard directly without exposing file-based URL.


M) assets/favicon.svg
Where used:
- Browser tab icon for dashboard.

Main logic:
- Neural-network inspired vector icon with gradient.

Why used:
- Branding and professional site identity.


N) README.md
Where used:
- Repository documentation page.

Main logic:
- Contains banner, logo, badges, module summary, structure, and live link.

Why used:
- Improves project presentation and onboarding for viewers.
